XMPro DocFX Documentation Project Instructions

Project Overview

This is the XMPro public documentation site that has been migrated from GitBook to DocFX. The documentation is built using DocFX static site generator and deployed to multiple version-specific GitHub repositories.

Project Architecture

  • Documentation content: docs/ directory
  • Images: docs/images/ (must use kebab-case naming)
  • Configuration: docfx.json (DO NOT modify sitemap.baseUrl)
  • Navigation: toc.yml files
  • Templates: xmpro-template/public/ (minimal CSS/JS only)
  • Generated site: _site/ (never edit directly)

Coding Standards

Image Naming

  • Use kebab-case for all image filenames: configure-agent-dialog.png
  • Never use spaces, parentheses, or uppercase in image names
  • Store all images in docs/images/ directory
  • Reference images with relative paths: ![Alt text](../../images/image-name.png)

URL Handling

  • Never hardcode version numbers in URLs (no v4.4, v4.5)
  • Use relative paths for all internal references
  • Let DocFX handle URL generation through configuration

DocFX Alert Syntax

  • Use > [!NOTE] for general information
  • Use > [!TIP] for helpful tips
  • Use > [!WARNING] for important warnings
  • Use > [!CAUTION] for critical warnings
  • Use > [!IMPORTANT] for important information

Testing Requirements

  • Always run docfx --serve locally before committing
  • Verify no build warnings or errors
  • Check all images load correctly (no 404s)
  • Test navigation changes work properly

Common Workflows

Adding Documentation Pages

  • Create markdown file in appropriate docs/ subdirectory
  • Add optional YAML front matter for title and description
  • Update corresponding toc.yml file
  • Test with docfx --serve before committing

Managing Images

  • Save with kebab-case names to docs/images/
  • Update markdown references to use relative paths
  • Remove unused images to keep repository clean
  • Verify all image references resolve correctly

Navigation Updates

  • Edit toc.yml files for menu structure
  • Use proper YAML syntax with name and href properties
  • Support nested items for hierarchical navigation
  • Test navigation changes thoroughly

@CONTRIBUTING.md

Deployment Process

  • Pipeline builds documentation once
  • Deploys to multiple version repositories in parallel
  • Never modify deployment scripts without understanding multi-version process
  • Changes auto-deploy to public GitHub Pages sites

Style Guidelines

  • Use descriptive commit messages starting with docs:
  • Follow existing documentation patterns
  • Maintain kebab-case image naming strictly
  • Keep all URLs version-agnostic
  • Test thoroughly before committing

Common Commands

# Build and serve documentation locally
cd public-documentation
docfx --serve

# Build without serving
docfx

# Clean build
rm -rf _site && docfx
