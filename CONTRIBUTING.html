XMPro Documentation Contributor Guide

Quick Start

  1. Install DocFX: dotnet tool install -g docfx
  2. Clone the repository and checkout a new branch
  3. Make your changes:
    • Add new pages in appropriate folders
    • Edit existing markdown files
    • Add images to /docs/images/
  4. Preview locally: 
    cd public-documentation
docfx --serve
  5. Verify at http://localhost:8080
  6. Commit with descriptive message
  7. Create a pull request

Introduction

This guide outlines the process for contributing to the XMPro documentation. Whether you're adding new content, updating existing pages, or fixing issues, this guide will help you maintain consistency and quality across the documentation.

Prerequisites

  1. Git - For version control
  2. Text Editor - VS Code, Sublime, or any markdown editor
  3. Web Browser - For testing the migrated documentation
  4. Access to the Repository - Clone permissions for the XMPro documentation repository
  5. DocFX - Documentation generation tool (see installation below)
  6. .NET SDK - Required for DocFX (version 6.0 or later)

Installing DocFX

DocFX is the tool that converts our markdown files into a static website. You need to install it to preview and build the documentation locally.

Install as .NET Tool (Recommended)

# Install DocFX globally
dotnet tool install -g docfx

# Verify installation
docfx --version

Setup

1. Repository Setup

# Clone the repository
git clone [repository-url]
cd xmpro-development-ubuntu

# Checkout the migration branch
git checkout 20290-migrate-public-docs

# Navigate to documentation folder
cd public-documentation

2. DocFX Local Development

Building the Documentation

To build the documentation without serving it:

# Build the documentation
docfx docfx.json

# This creates the static site in the _site directory
# Useful for checking build errors without starting a server

Serving the Documentation Locally

To build and serve the documentation with live preview:

# Build and serve with auto-refresh
docfx docfx.json --serve

# The site will be available at http://localhost:8080
# The console will show build output and any errors

Alternative port: If port 8080 is in use:

# Serve on a different port
docfx docfx.local.json --serve -p 8081

Development Workflow

  1. Start the DocFX server in one terminal:

    cd public-documentation
docfx docfx.json --serve

    Keep this terminal open to see build errors and warnings.

  2. Make your changes in another terminal or editor:

    • Edit markdown files
    • Add images
    • Update navigation

  3. Rebuild to see changes:

    • DocFX does NOT auto-rebuild on file changes
    • You must manually rebuild after making changes
    • In the DocFX terminal, press Ctrl+C to stop
    • Run docfx docfx.json --serve again
    • Refresh your browser

Common DocFX Commands

# Build only (no server)
docfx

# Build and serve
docfx --serve

# Build with verbose logging (helpful for debugging)
docfx -v

# Clean build (removes _site directory first)
rm -rf _site && docfx

# Serve on custom port
docfx --serve -p 3000

Contribution Workflow

Step 0: Verify DocFX is Working

Before starting any documentation work, ensure DocFX is properly set up:

# Navigate to the documentation folder
cd public-documentation

# Build and serve the documentation
docfx --serve

# Open http://localhost:8080 in your browser
# You should see the XMPro documentation site

If you encounter errors:

  • Check that DocFX is installed: docfx --version
  • Ensure you're in the correct directory
  • Look for error messages in the console output

1. Create a New Branch

Before starting work:

# Ensure you're on the main branch and up to date
git checkout main
git pull origin main

# Create a new branch for your work
git checkout -b docs/your-feature-name
# Examples:
# git checkout -b docs/add-ai-agent-guide
# git checkout -b docs/update-installation-steps
# git checkout -b docs/fix-broken-images

2. Making Changes

Adding a New Page

  1. Choose the right location:

    • /docs/getting-started/ - For onboarding and setup guides
    • /docs/how-tos/ - For step-by-step tutorials
    • /docs/concepts/ - For explanations of concepts and features
    • /docs/blocks-toolbox/ - For documentation about specific blocks

  2. Create the markdown file:

    # Example: Creating a new how-to guide
touch docs/how-tos/configure-ai-agents.md

  3. Add front matter (optional but recommended):

    ---
title: How to Configure AI Agents
description: Learn how to set up and configure AI agents in XMPro
---

  4. Write your content using proper markdown and DocFX features

  5. Add to table of contents (toc.yml or section-specific toc.yml):

    - name: Configure AI Agents
  href: configure-ai-agents.md

Editing Existing Pages

  1. Locate the file you want to edit in the /docs/ directory

  2. Make your changes:

    • Fix typos or grammatical errors
    • Update outdated information
    • Add new sections or examples
    • Improve clarity or formatting

  3. Follow the style guide:

    • Use clear, concise language
    • Include code examples where helpful
    • Add screenshots for UI-related content
    • Keep paragraphs short and scannable

Adding Images

  1. Name your images using kebab-case:

    • ✅ configure-ai-agent-dialog.png
    • ✅ data-stream-overview.jpg
    • ❌ Configure AI Agent Dialog.png
    • ❌ DataStreamOverview.jpg

  2. Save images to /docs/images/

  3. Reference in markdown:

    ![Configure AI Agent Dialog](../../images/configure-ai-agent-dialog.png)

  4. Add captions when helpful:

    ![Configure AI Agent Dialog](../../images/configure-ai-agent-dialog.png)
*Figure 1: The AI Agent configuration dialog*

3. Test Your Changes

Build and Preview

  1. Rebuild DocFX:

    cd public-documentation
# Stop the current server if running (Ctrl+C)
# Rebuild and serve
docfx --serve

  2. Check the console for:

    • Build errors (red text) - Must fix before committing
    • Warnings (yellow text) - Should fix if possible
    • Missing file references
    • Invalid cross-references

  3. Test in browser:

    • Navigate to http://localhost:8080
    • Find your new or edited page(s)
    • Verify all images load (check for 404s)
    • Test all links work correctly
    • Check video embeds play
    • Ensure formatting looks good
    • Test on different screen sizes

  4. Quality checklist:

    • [ ] Content is accurate and up-to-date
    • [ ] Images display correctly
    • [ ] All links work (internal and external)
    • [ ] Code blocks have proper syntax highlighting
    • [ ] Notes/alerts use appropriate types
    • [ ] Navigation includes your new page (if applicable)
    • [ ] No spelling or grammar errors
    • [ ] Follows the documentation style guide

Debugging Build Errors

If DocFX reports errors:

# Get more detailed error information
docfx docfx.json -v

# Common errors and fixes:
# "Invalid cross reference" - Check the link syntax
# "File not found" - Verify file paths and names
# "Invalid metadata" - Check YAML front matter

4. Commit Changes

  1. Stage your changes:

    git add docs/path/to/your-file.md
git add docs/images/your-new-images.png
git add toc.yml  # If you modified it

  2. Write a clear commit message:

    git commit -m "docs: Add guide for configuring AI agents

- Added comprehensive setup instructions
- Included troubleshooting section
- Added screenshots for each configuration step"

    Commit message format:

    • Start with docs: for documentation changes
    • Use present tense ("Add" not "Added" in the subject)
    • Keep the first line under 50 characters
    • Add details in the body if needed

5. Create a Pull Request

  1. Push your branch:

    git push origin docs/your-feature-name

  2. Create the PR on GitHub/GitLab:

    • Give it a descriptive title
    • Fill out the PR template (if available)
    • Link any related issues
    • Request review from appropriate team members

  3. PR description should include:

    • What changes you made
    • Why the changes were needed
    • Any special testing instructions
    • Screenshots if UI-related

### Prerequisites
- Admin access to XMPro
- SMTP server details
- Valid email addresses

### Steps

1. **Navigate to Settings**
   
   Click on the settings icon in the top navigation bar.
   
   ![Settings Icon](../../images/settings-icon.png)

2. **Select Email Configuration**
   
   From the settings menu, choose "Email Configuration".
   
   > [!NOTE]
   > You need administrator privileges to access this section.

3. **Enter SMTP Details**
   
   Fill in the following fields:
   - **Server**: Your SMTP server address
   - **Port**: Usually 587 for TLS or 465 for SSL
   - **Username**: Your SMTP username
   - **Password**: Your SMTP password
   
   ```json
   {
     "server": "smtp.example.com",
     "port": 587,
     "username": "notifications@example.com",
     "enableSSL": true
   }

  1. Test the Configuration

    Click "Send Test Email" to verify your settings.

    Tip

    Check your spam folder if the test email doesn't arrive.

Troubleshooting

DocFX Issues

  1. DocFX Command Not Found

    # Verify installation
dotnet tool list -g

# Reinstall if needed
dotnet tool install -g docfx

# Or check PATH if manually installed
echo $PATH

  2. Port Already in Use

    # Use a different port
docfx --serve -p 8081

# Or find and kill the process using port 8080
# On Linux/Mac:
lsof -i :8080
kill -9 [PID]

# On Windows:
netstat -ano | findstr :8080
taskkill /PID [PID] /F

  3. Build Hangs or Is Slow

    • Clean the output directory: rm -rf _site
    • Check for circular references in toc.yml
    • Reduce console verbosity if too much output

Common Issues and Solutions

404 Image Errors

  • Problem: Images not loading in the preview
  • Solution:
    1. Check image filename uses kebab-case
    2. Verify image is in /docs/images/
    3. Check the path in your markdown is correct
    4. Remember: all image paths should be ../../images/filename.ext

Broken Internal Links

  • Problem: Links to other pages return 404
  • Solution:
    1. Use absolute paths from docs root: /docs/path/to/page.md
    2. Ensure the target file exists
    3. Check for typos in the path
    4. Don't include .html extension in links

Build Warnings

  • Problem: DocFX shows warnings during build
  • Common warnings and fixes:
    • Invalid cross reference - Fix the link syntax
    • File not found - Check file paths and names
    • Duplicate uid - Ensure unique identifiers
    • Invalid metadata - Fix YAML front matter

Formatting Issues

  • Problem: Content doesn't display as expected
  • Solutions:
    • Ensure proper markdown syntax
    • Check for unclosed HTML tags
    • Verify code blocks have proper language tags
    • Use preview to catch issues early

Best Practices

Writing Style

  1. Be Clear and Concise

    • Use simple, direct language
    • Avoid jargon unless necessary
    • Define technical terms on first use
    • Keep sentences and paragraphs short

  2. Be Consistent

    • Use the same terminology throughout
    • Follow existing naming conventions
    • Maintain consistent formatting
    • Use the same voice and tone

  3. Be Helpful

    • Anticipate user questions
    • Provide examples and screenshots
    • Include troubleshooting sections
    • Link to related topics

Content Guidelines

  1. Structure Your Content

    • Start with an overview
    • Use descriptive headings
    • Break complex topics into sections
    • Include a summary for long articles

  2. Use Visual Aids

    • Add screenshots for UI instructions
    • Include diagrams for complex concepts
    • Use code examples liberally
    • Annotate images when helpful

  3. Make it Scannable

    • Use bullet points and numbered lists
    • Bold important terms
    • Include a table of contents for long pages
    • Use tables for comparing options

Technical Writing Tips

  1. For How-To Guides

    • Start with prerequisites
    • Number each step
    • Include expected outcomes
    • Add troubleshooting section

  2. For Concept Docs

    • Explain the "why" before the "how"
    • Use analogies when helpful
    • Include real-world examples
    • Link to related how-tos

  3. For Reference Docs

    • Use consistent structure
    • Include all parameters/options
    • Provide examples for each
    • Keep descriptions brief

Image Guidelines

  1. Screenshots

    • Capture at reasonable resolution
    • Highlight important areas
    • Crop to focus on relevant parts
    • Use consistent window sizes

  2. Diagrams

    • Keep them simple and clear
    • Use consistent colors/styles
    • Include a legend if needed
    • Save as PNG for quality

  3. Naming

    • Use descriptive names
    • Include feature/page context
    • Always use kebab-case
    • Add version numbers if needed
