XMPro Documentation Deployment

Overview

The XMPro documentation pipeline uses a stage-based approach with manual approvals, giving you complete control over which version repositories receive updates.

How It Works

File: azure-pipelines.yml

Pipeline Features

  • Stage-based deployment with independent approvals
  • Each version has its own deployment stage
  • Manual approval required for each deployment
  • Clear deployment tracking and history
  • 24-hour approval timeout for safety

Pipeline Stages

The pipeline consists of these stages:

Build Documentation ✓
    ├─> Deploy to v4.4 (latest) [Awaiting Approval]
    └─> Deploy to v4.5 [Awaiting Approval]
            └─> Deployment Summary

Each deployment stage can be independently approved or rejected.

Usage

Running the Pipeline

When you trigger the pipeline, it will:

  1. Build the documentation automatically
  2. Present deployment stages for approval
  3. Wait for manual approval (up to 24 hours)
  4. Deploy only to approved stages
  5. Summarize the results

Deployment Scenarios

Deploy to v4.4 Only

  • Click "Resume" on the v4.4 approval
  • Click "Reject" on the v4.5 approval (or let it timeout)

Perfect for: Updates to the current stable version

Deploy to Both Versions

  • Click "Resume" on both v4.4 and v4.5 approvals

Perfect for: Bug fixes and documentation improvements that apply to both versions

Deploy to v4.5 Only

  • Click "Reject" on the v4.4 approval (or let it timeout)
  • Click "Resume" on the v4.5 approval

Perfect for: v4.5-specific features or preview documentation

Build Only (No Deployment)

  • Click "Reject" on both approvals (or let them timeout)

Perfect for: Testing build without deployment

Deployment Process

  1. Trigger Pipeline

    • Via Azure DevOps UI
    • No parameters needed - just run the pipeline

  2. Build Stage

    • Documentation is built
    • Validation checks run
    • Build artifacts created

  3. Approval Stages (one per version)

    • Each deployment stage pauses for manual approval
    • Approvers see deployment target information
    • Click "Resume" to deploy or "Reject" to skip
    • ⏰ Timeout: 24 hours - If no action is taken, the deployment is automatically rejected

  4. Deploy Stages

    • Deploys only to approved repositories
    • Creates deployment tracking files
    • Updates git history

  5. Summary Stage

    • Runs after all deployments complete
    • Provides guidance on viewing results

Running the Pipeline

Via Azure DevOps UI

  1. Navigate to Pipelines
  2. Find the documentation pipeline
  3. Click "Run pipeline"
  4. Click "Run" (no parameters to configure)
  5. Wait for build to complete
  6. Approve desired deployment stages

Via Azure CLI

# Run the pipeline
az pipelines run --name "XMPro Documentation" --branch main

# Note: Approvals must be done through the Azure DevOps UI

Best Practices

  1. Select Appropriate Versions

    • New features → Latest only
    • Bug fixes → All affected versions
    • Version-specific → That version only

  2. Approval Process

    • Review all changes before clicking "Resume"
    • Verify deployment targets are correct
    • Click "Reject" to skip unwanted deployments

  3. Test Before Deploying

    cd public-documentation
docfx --serve
# Review at http://localhost:8080

Advantages

  • Complete Control: Choose exactly where changes go
  • Maximum Safety: Mandatory approval prevents accidental deployments
  • Flexibility: Different strategies for different changes
  • Visibility: Clear record of deployments
  • Simplicity: Only two checkboxes to configure
  • Accountability: Every deployment is reviewed and approved

Troubleshooting

No Parameters Shown

  • Ensure you're running the correct pipeline
  • Check Azure DevOps permissions

Approval Stage Not Triggered

  • Ensure at least one deployment target is selected
  • Check pipeline logs for errors

Approval Timeout

  • Default timeout: 24 hours
  • What happens: Pipeline shows as "Failed" if no action taken
  • This is normal: Timeout = automatic rejection for safety
  • To avoid: Ensure approvers are available before running pipeline
  • To re-run: Simply trigger the pipeline again with same parameters

Deployment Fails

  • Check PAT token permissions
  • Verify repository access
  • Review pipeline logs

This manual approach ensures documentation changes are deployed intentionally to the correct versions with appropriate oversight.
